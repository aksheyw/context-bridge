# Handoff prompt template

> The artifact that turns a closed Claude Code session into a warm next-session start. Generated by `/cb-handoff` (standalone) or as step 10 of `/cb-save-sync`.

> **Provenance:** original template by Akshey Walia, evolved across multiple of the author's projects through trial and error. See [CREDITS.md](../../CREDITS.md) §3.3.

---

## The canonical prompt (what the user pastes next session)

```
I'm continuing the <project-name> work. Last session closed <YYYY-MM-DD>.

Read these in order, fast:
1. .claude/wiki/_hot.md
2. .claude/wiki/_log.md (most recent session entry only)
3. .claude/wiki/_findings.md
4. SESSION_HANDOFF_<YYYY-MM-DD>.md (this file — full handoff context)

Then summarize back to me in 5 bullets:
- What we're building + who it's for + who it's NOT for
- What's pending and priority (top 3-5 items)
- Top 3 risks
- Today's first task — a concrete 60-min plan
- The killer demo or output for the next session

After your summary, ask me:
1. <question 1 — pulled from _hot.md "Open blockers / questions">
2. <question 2 — pulled from _hot.md, or a scope question if only one open>
3. Co-pilot mode: A (standby) / B (active co-pilot) / C (code reviewer) / D (build for me)?

Honesty: 95% confidence gate. Never fabricate paths or commands. Use Read/Grep/Glob to verify.
Velocity: speed + usefulness wins. Cut scope when a gate is at risk.
```

## Why this shape

### Read 4 small files, not 20 large ones
Three wiki files + one handoff doc. All are small (`_hot.md` ~50 lines, `_log.md` latest entry ~30 lines, `_findings.md` 1-page summary, handoff doc ~150-250 lines). Loading them costs <2k tokens. Loading the whole codebase costs 20k+ before any work starts.

### Force a 5-bullet summary back
The summary back is the verification — if the model can't summarise the state in 5 bullets, the wiki is too thin or the model didn't actually read. Either is recoverable; both should surface BEFORE work starts.

### "What we're building + who it's NOT for"
Forcing the non-goals into the summary catches scope drift the moment it appears. If the model claims context-bridge is "for teams" or "fixes mid-session bloat", the user catches it in bullet 1 — not after 3 hours of off-spec work.

### Ask back 1-3 questions
- Question 1 from `_hot.md` open blockers — the wiki said this needs a decision; the user is in the chair, decide it now.
- Question 2 from `_hot.md`, or a scope question on the top task — directional alignment before any code.
- Co-pilot mode — explicit collaboration shape (standby / co-pilot / reviewer / autonomous build). Sets expectations for the session.

### Honesty + velocity reminders at the bottom
- Honesty: 95% gate + never-fabricate. The wiki itself encodes these in CLAUDE.md but the prompt reinforces.
- Velocity: cut scope when a gate is at risk. Prevents the "spec said 10 things, I'll do all 10" failure mode.

## Adaptation rules

### Always preserved (changing breaks the warm-resume guarantee)
- Read-in-order list of the 4 files.
- 5-bullet summary back requirement.
- Question gate at the end.
- Honesty + velocity reminders.

### Project-specific (encouraged to customise)
- The 4th file's name (handoff filename varies by date).
- The 5-bullet contents (each project's "what we're building" differs).
- The 1-3 questions (always pulled fresh from `_hot.md`).
- Co-pilot mode options (A/B/C/D is one shape; pick what works).

### Anti-patterns (don't do these)
- ❌ Loading the entire wiki "for context". Defeats the point.
- ❌ Adding "review my recent commits" — the wiki + handoff already summarise.
- ❌ Skipping the question gate. The questions are how human judgment stays in the loop.
- ❌ Inventing questions when `_hot.md` is empty. Honest: ask the user what they want to focus on.

## The full handoff document

The prompt is the FIRST section of `SESSION_HANDOFF_<today>.md`. The doc also contains:

1. **Metadata header** (closed date, phase, branch + commit).
2. **The prompt** (verbatim, fenced for easy copy).
3. **TL;DR for future-me** (2-3 sentences).
4. **What was done this session** (bulleted, from `_log.md` latest entry).
5. **What's pending — prioritized** (P0/P1/P2 from `_hot.md`).
6. **Open findings register** (severity-ordered table from `_findings.md`, OPEN + IN PROGRESS only).
7. **Skills + agents likely useful next session** (3-5 conservative picks).
8. **Cross-references** (repo paths, wiki paths, prior handoffs).

Target: 150-250 lines total. Omit sections with no content — don't print `(none)` placeholders.

After your summary, ask me:

## Failure modes

| Scenario | What to do |
|---|---|
| `_hot.md` is empty | Generate the prompt with `(none yet)` in the open-questions slot; user answers from intent. |
| No prior handoff exists | First handoff is fine; the 4th read just doesn't exist yet (user pastes prompt, model reads only files 1-3). |
| Wiki has secrets in it | `/cb-handoff` re-runs the step-0 scan on the generated doc; blocks if hit. |
| Multiple handoffs same day | Filename appends `-HHMM`. Never overwrites. |
| Project doesn't use `_findings.md` | Skip that bullet in the summary-back request; don't ask for what isn't there. |
