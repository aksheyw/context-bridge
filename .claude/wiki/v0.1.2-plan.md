# v0.1.2 Plan — Phased Best-Practice Gaps to Address

Captured at Session 5 close (2026-05-26). v0.1.0 + v0.1.1 shipped. This doc enumerates the gaps between context-bridge and industry-standard OSS practice, ranked by impact-vs-effort and assigned to explicit phases.

**These are gaps, not bugs.** v0.1.1 is correct and complete for its declared scope. The items below are improvements — adopting them strengthens trust and reduces friction, but skipping any of them doesn't break the product.

---

## TIER 1 — v0.1.2 polish (~1 hr total, queued for next session)

High signal, low effort. Should ship before any public announcement. None of these change skill behavior; all are infrastructure / hygiene.

| # | Gap | Phase | Effort | Detail |
|---|---|---|---|---|
| T1-1 | **`shellcheck` not in CI for `.githooks/pre-commit`** | v0.1.2 | 20 min | `bash -n` only catches syntax. shellcheck catches semantic bugs (same class as the `grep -c` returning `"0\n0"` bug fixed in `verify.sh` during v0.1.1). Add `shellcheck .githooks/pre-commit` step to `pii-scrub-check.yml`. |
| T1-2 | **CI matrix: ubuntu-latest only (macOS bash-3.2 claim unverified)** | v0.1.2 | 5 min YAML | Hook *claims* macOS bash-3.2 portability in its comments. No proof. Add `strategy.matrix.os: [ubuntu-latest, macos-latest]` to the workflow. |
| T1-3 | **No GitHub Release for v0.1.0 or v0.1.1 (tags exist; Releases page empty)** | v0.1.2 | 10 min | `gh release create v0.1.0 --notes-file <text>` and same for v0.1.1. Otherwise GitHub UI says "no releases yet" — looks abandoned. |
| T1-4 | **GitHub Discussions not enabled** | v0.1.2 | 1 click | Better forum for "how do I" than issues. Settings → Features → Discussions ON. Keeps issue tracker clean for actual bugs. |
| T1-5 | **No branch protection on `main`** | v0.1.2 | 2 min | Require `pii-scrub-check` to pass before merge. Settings → Branches → Add rule. Solo project, but compounds discipline. |
| T1-6 | **No `.github/FUNDING.yml`** | v0.1.2 | 5 min | Enables GitHub Sponsor button. CREDITS.md already encourages sponsoring Vincent for Superpowers; if context-bridge attracts the same audience, set up the path. |
| T1-7 | **No `.github/CODEOWNERS`** | v0.1.2 | 2 min | Documents review responsibility. Even solo, it's clarity. `* @aksheyw` is the v0.1.2 line. |

**v0.1.2 ship blocker:** none of these are required for correctness. Ship-bar is "all 7 done OR explicit acceptance of which we deferred and why."

## TIER 2 — Worth doing before announcing publicly (1-3 hrs each)

Quality + supply-chain practices. Skipping doesn't break anything but adopters at orgs may want some of these before they trust the skill.

| # | Gap | Phase | Effort | Detail |
|---|---|---|---|---|
| T2-1 | **No tests for `.githooks/pre-commit`** | v0.2 OR post-day-30 retro | 1-2 hr | bats-core fixtures: file containing entropy-required real-shaped secret (must block), file containing bare prefix (must pass), clean file (must pass). Right now hook correctness is asserted, not tested. |
| T2-2 | **No CodeQL static analysis** | v0.2 | 10 min setup | Free GitHub Advanced Security feature. Catches issues in workflow YAML + any future code. Zero ongoing cost. |
| T2-3 | **No Dependabot config** | v0.2 | 5 min | We have no deps today. The moment we pin an action version, Dependabot keeps it patched. Forward-looking. |
| T2-4 | **No architecture diagram (Mermaid)** | v0.2 | 30 min | One picture: `SKILL.md` (entry) → `commands/` → `templates/` (lazy) → `references/` (on-demand) → adopter's `.claude/wiki/`. Adopters internalize the structure in 10s. |
| T2-5 | **No social preview image (GitHub OG)** | v0.2 | 15-30 min | When repo shared on LinkedIn/Twitter, default GitHub OG card is empty. 1280×640 image set in Settings → Social preview. Controls what's rendered when linked externally. |
| T2-6 | **No issue labels configured** | v0.2 | 10 min | `bug`, `feature`, `attribution`, `good-first-issue`, `help-wanted`, `question`, `docs`. Labels make tracker scannable. |
| T2-7 | **No `AGENTS.md` at repo root** | v0.2 | 20 min | Newer convention for repos consumed by AI agents (Codex / Cursor / future tools). Machine-readable form of "what this skill is + how to use it". Cheap cross-tool insurance. |

## Explicit NON-goals (industry practices we will NOT adopt)

Honest exclusion list. These are common in larger OSS but would be wrong for context-bridge's scope.

| # | Practice | Why we won't do it |
|---|---|---|
| NG-1 | Telemetry, even opt-in | The skill claims privacy via no-runtime-services. Adding analytics later breaks the promise. Adoption tracked via stars / clones / DMs per `docs/success-criteria.md`. |
| NG-2 | Web UI for the wiki | Out of scope per spec §11.3. Dilutes the "small, focused, single-tool" pitch. |
| NG-3 | Plugin marketplace listing | Premature; wait for day-30 retro decision-branch. |
| NG-4 | Discord / Slack community | Solo-maintainer + unknown adoption. Empty Discord worse than no Discord. Discussions covers the need. |
| NG-5 | Custom domain / standalone website | README + docs/ on GitHub is the right surface. Domain costs $12/yr; benefit at this scale: zero. |
| NG-6 | Newsletter / mailing list | Same logic as Discord. |
| NG-7 | Automatic version bumping on every push | Conventional commits + manual release is fine until cadence proves otherwise. |
| NG-8 | A "context-bridge ecosystem" of plugins | v0.1 scope is intentionally narrow. Don't invite extensions until v0.2 ships and proves value. |

## Strengths to PRESERVE (do not refactor away in pursuit of "best practice")

Things v0.1.1 does that are genuinely better than typical OSS practice. Don't sacrifice these for any of the Tier 1-3 items above.

- **Attribution rigor** — Karpathy gist linked everywhere + Vincent + Anthropic + Kent Beck. Rare for OSS this small.
- **Embedded honesty rules in adopters' CLAUDE.md** — genuinely novel; the discipline travels with the wiki.
- **14-lens deep-review pre-merge** — found 14 issues v0.1.0 would have shipped with.
- **Entropy-required secret patterns** — industry standard but rarely done correctly.
- **Sub-200-line SKILL.md (size budget enforced in CI)** — self-discipline matching the problem the skill solves.
- **Single-command local gate runner (`scripts/verify.sh`)** — better than the "run these 7 commands" pattern most projects use.
- **Worked example with 3 session snapshots** — better than 90% of OSS-skill-style repos.
- **Honest non-goals + when-not-to-use doc** — most projects oversell; this one explicitly disqualifies.
- **CI/local gate parity** — what `verify.sh` runs locally is what `pii-scrub-check.yml` runs in CI. No drift surface.
